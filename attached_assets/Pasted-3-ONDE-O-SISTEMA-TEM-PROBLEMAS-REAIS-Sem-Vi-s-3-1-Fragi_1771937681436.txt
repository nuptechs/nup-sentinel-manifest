3. ONDE O SISTEMA TEM PROBLEMAS REAIS (Sem Viés)
3.1. Fragilidade Estrutural do Frontend Analyzer
O frontend-analyzer.ts tem 5.864 linhas em um único arquivo. Isso é um problema sério de manutenção e testabilidade. Comparando com tools de análise estática maduros:

Semgrep: Core do parser é ~200 LOC por regra, com separação clara
ESLint: Cada regra é um módulo isolado e testável
O monólito atual mistura parsing, symbol resolution, graph building, event graph, state flow, e HTTP resolution tudo junto. Qualquer mudança tem risco de efeito colateral.

3.2. Análise Sem Type System
O TypeScript Compiler API dá acesso ao type checker (ts.createProgram + checker.getTypeAtLocation), mas o PermaCat não usa. Ele trabalha com ts.createSourceFile (sem type information). Isso significa:

Não resolve types de retorno de funções
Não consegue seguir chamadas via interfaces/types
Não resolve generics (ex: HttpClient<T>)
Não distingue api.get() de cache.get() semanticamente
Isso é a raiz de porque 307 interações não são resolvidas — o sistema tenta compensar com heurísticas (nome de imports, padrões de nome), mas isso é fundamentalmente frágil.

Porém: Usar ts.createProgram requer resolver módulos, o que precisa de node_modules e tsconfig.json. Em análise de ZIP uploads, isso é complexo.

3.3. Acoplamento Frontend ↔ Backend Via Strings
A conexão frontend→backend é feita por URL matching (matchUrlToEndpoint). Isso funciona para REST clássico, mas quebra em:

GraphQL: URL única (/graphql), operação no body
gRPC-Web: Protocol buffers, não HTTP methods
tRPC: Endpoints type-safe via RPC, não REST
Server Actions (Next.js): Sem URL explícita
WebSocket-based APIs: Sem HTTP method
3.4. Inference Engine é Útil Mas Conceitualmente Limitado
O frontend-inference-engine.ts faz inferências inteligentes (URL → controller, entity, roles), mas:

São heurísticas, não análise real
/api/v2/user-settings vira UserSettingController com entity UserSetting, o que pode ou não ser correto
A inferência de roles (/admin/* → ADMIN) é simplista demais para produção
O sistema mistura dados reais (extraídos do código) com dados inferidos (adivinhados). O manifest não diferencia claramente os dois, o que é problemático para auditoria.

3.5. Completeness Score é Misleading
O endpointResolution calcula entries com endpoint / total entries. Mas como mostrado na análise, ~183 das 307 entries sem endpoint são state setters legítimos (setState, toggleModal, etc.) que nunca terão endpoint. Medir sua ausência como "incompleteness" é incorreto.

O score deveria ser: entries HTTP resolvidas / entries que DEVERIAM ter HTTP. Isso requer classificar antes de medir.

3.6. Sem Testes Automatizados
Não encontrei nenhum arquivo de teste. Para um tool de análise estática, isso é um risco crítico. Cada mudança no analyzer pode quebrar silenciosamente cenários anteriores. A verificação manual (PetClinic + AIM) é necessária mas insuficiente.
